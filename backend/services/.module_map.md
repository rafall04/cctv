# backend/services/.module_map.md

## Service Groups

- Camera domain:
  - `cameraService.js`: CRUD, import/export, restore, delivery normalization, bulk area operations, read-model caching.
  - `cameraSourceLifecycleService.js`: source/IP/codec/transport/enabled update classification, MediaMTX refresh orchestration, runtime reconnecting state, stream revision bumps, and masked lifecycle event writes.
  - `cameraHealthService.js`: probe scheduling, runtime health scoring, online/offline transitions, recording/thumbnail coordination.
  - `cameraHealthPolicy.js`: pure external health policy resolution.
  - `cameraMonitoringAlertPolicy.js`: pure Telegram monitoring alert policy for opt-in strict internal HLS monitoring eligibility and online/offline transition detection.
  - `telegramAlertConfirmationPolicy.js`: pure Telegram alert confirmation timing policy that delays DOWN/UP sends until monitored state remains stable.
  - `cameraRuntimeStateService.js`: runtime state persistence and signal tracking.
- Streaming domain:
  - `mediaMtxService.js`: MediaMTX path sync and API interaction.
  - `streamService.js`: stream URL/status operations.
  - `streamWarmer.js`: proactive stream warmup.
- Recording domain:
  - Facade & top-level orchestration:
    - `recordingService.js`: thin facade exposing `startRecording`, `stopRecording`, `restartRecording`, `reconcileRecordingLifecycle(All)`, `autoStartRecordings`, `getRecordingStatus`, `getStorageUsage`, `attachScheduler`, `initializeBackgroundWork`, `shutdown`. Internals delegate to focused services; do not add new responsibilities here.
  - Paths, intervals & policy (pure):
    - `recordingPaths.js`: single source of truth for `RECORDINGS_BASE_PATH` (ENV override via `RECORDINGS_DIR`). Every recording service reads from here.
    - `recordingIntervalsPolicy.js`: every interval, threshold, batch size, and cooldown for the recording domain. ENV-overridable knobs: `RECORDING_SCHEDULED_CLEANUP_INTERVAL_MS`, `RECORDING_RECOVERY_MAX_CONCURRENT`, `RECORDING_EMERGENCY_DISK_THRESHOLD_BYTES`, `RECORDING_EMERGENCY_DISK_TARGET_BYTES`.
    - `recordingLifecyclePolicy.js`: pure desired-state decision (start / stop_offline / stop_disabled / stop_unrecordable / wait_cooldown / noop_*).
    - `recordingPartialRecoveryPolicy.js`: pure retry/quarantine decision for recovery.
    - `recordingPathSafetyPolicy.js`: pure path/filename/byte-range safety shared by playback, cleanup, recovery.
    - `recordingSegmentFilePolicy.js`: pure path/name classifier for final/partial/temp + `resolveSegmentSource(basePath, cameraId, filename) → { sourceType, sourcePath, finalFilename }`.
    - `recordingTimePolicy.js` / `recordingProcessTimePolicy.js`: app-timezone filename parsing and FFmpeg `TZ` env.
    - `recordingRetentionPolicy.js`: retention window + safe-filename + per-file delete decision (re-exports intervals constants for back-compat).
    - `recordingFailureClassifier.js`: classify FFmpeg exit into operator-meaningful lifecycle/failure reasons.
  - Lifecycle & process control:
    - `recordingLifecycleReconciler.js`: DB/process desired-state safety net. Single source for start/stop/`stop_disabled`/`stop_unrecordable` transitions; called by health monitor tick, `autoStartRecordings`, `updateRecordingSettings`, and `cameraHealthService` online/offline transitions.
    - `recordingProcessManager.js`: FFmpeg start/stop/restart serialization. Owns spawn + graceful kill.
    - `recordingRuntimeState.js`: in-memory FFmpeg process state.
    - `recordingHealthMonitor.js`: per-camera stream health (data freshness, cooldown, suspend reason, restart-on-freeze). Owns the 5s tick loop. Internal cross-method calls go through a late-binding `api` object so `vi.spyOn` works.
    - `recordingStarter.js`: pure builder for `sourceConfig`, `ffmpegArgs`, `spawnOptions`, and camera/pending directories. `prepareRecordingStart({ camera, recordingsBasePath })` returns a struct the facade hands to `recordingProcessManager.start()`.
    - `recordingStderrParser.js`: pure FFmpeg stderr line classifier (`segment_completed` / `segment_debug` / `error` / `other`).
    - `recordingAutoStarter.js`: one-shot boot auto-start. Pre-marks offline cameras as suspended, then delegates to lifecycle reconciler.
  - Segments, recovery, repositories:
    - `recordingSegmentFinalizer.js`: idempotent finalization pipeline (validate duration → remux to temp → atomic promote → DB upsert → cleanup source). `drain(timeoutMs)` loops until in-flight is empty or deadline hits.
    - `recordingRecoveryService.js`: bounded recovery queue with single public entrypoint `enqueueRecovery(input) → Promise<result>`. `recoverNow` is no longer exported — every caller respects `maxConcurrent`. Owns `isFileOwned()` (the only in-progress tracker in the recording domain).
    - `recordingRecoveryScanner.js`: non-destructive folder scan for pending partials and final orphans. Exposes `scanOnce()` + `intervalMs`; scheduling is owned by the maintenance coordinator.
    - `recordingSegmentRepository.js`: duplicate-safe `(camera_id, filename)` persistence.
    - `recordingRecoveryDiagnosticsRepository.js`: operator-visible recovery states. `incrementAttempt` runs UPSERT+SELECT in a transaction so attempt_count reads are deterministic under concurrency.
  - Maintenance, cleanup, disk:
    - `recordingMaintenanceCoordinator.js`: owns lazy creation of `recordingRecoveryScanner`, `recordingBackgroundCleanupService`, `recordingEmergencyDiskService`. `registerSchedulerTasks(scheduler)` registers all 5 maintenance loops; `startLegacyTimers()` is the test-only manual scheduling fallback. Uses late-binding `api` object for spy compatibility.
    - `recordingCleanupService.js`: bounded per-camera cleanup + emergency cleanup. Owns per-camera concurrency lock AND rate-limit (`minIntervalMs`, default 60s). `drain(timeoutMs)` waits for in-flight cleanups.
    - `recordingBackgroundCleanupService.js`: slow background reconciliation of unregistered files (1 file per 10s). Exposes `buildQueue()`, `processOneQueueItem()`, intervals + initial delays.
    - `recordingEmergencyDiskService.js`: low-disk-space cleanup orchestration. Serializes `runEmergencyCheck` via internal in-flight flag.
    - `recordingDiskSpaceService.js`: cross-platform free-bytes reader behind injectable boundary.
    - `recordingFileOperationService.js`: the only safe delete/quarantine side-effect boundary for recording files.
    - `recordingMaintenanceStateRepository.js`: maintenance run status persistence.
  - Scheduling, playback, assurance:
    - `recordingScheduler.js`: `register({ name, task, intervalMs, initialDelayMs })` + `start()` / `stop()` + per-task telemetry (`runCount`, `lastRunAt`, `lastDurationMs`, `lastError`). Recording facade does not own raw timer handles in production.
    - `recordingPlaybackService.js`: segment lookup / playback policy / `updateRecordingSettings` → reconciler.
    - `recordingAssuranceService.js`: read-only assurance snapshot and operator-facing degradation summary.
    - `recordingHealthDashboardService.js`: read-only observability aggregator. `getSnapshot()` composes scheduler telemetry, recovery queue/diagnostics, recording process counts, restart history, and storage into one payload with a derived `ok`/`warning`/`critical` status. Backs `GET /api/admin/recording-health`. Each section is computed defensively so one failure degrades to a fallback instead of failing the endpoint.
    - `recordingHealthAlertService.js`: edge-triggered Telegram alerts. `checkAndAlert()` reads the health snapshot and sends a monitoring-chat message only when the level transitions (ok ⇄ warning ⇄ critical) — no re-spam while a level holds. Registered as the `recording-health-alert` scheduler task in server.js; gated by `config.telegram.healthAlertsEnabled` + `isTelegramConfigured()`.
  - Invariants (apply to all changes):
    - Recovery: playback only reads validated final MP4 rows from `recording_segments`; pending/partial/tmp/corrupt/orphan files are retained through retention+grace and reconciled before deletion.
    - Cleanup: all destructive paths flow through `recordingCleanupService.js` + `recordingRetentionPolicy.js`. Pending `.mp4.partial` recovery files from unstable connectivity must not be terminal-quarantined by recovery before retention cleanup owns their deletion. Final `.mp4` orphans can be quarantined only after retry exhaustion and must receive a recovery/finalizer pass before permanent deletion. Pending partials are retried with backoff and retained until retention+grace expires. Emergency disk cleanup uses the same safe-delete boundary and may bypass retention only for DB-registered segments below the emergency target.
    - In-progress tracking: `recoveryService.isFileOwned()` is the single source of truth. Do not introduce parallel `Set`s in callers.
    - Concurrency cap: every recovery path (segment-close, scanner, cleanup orphan, background cleanup) must call `enqueueRecovery` — never bypass the queue.
    - Lifecycle transitions: external services (`cameraHealthService`, `recordingPlaybackService.updateRecordingSettings`, boot auto-start) call `reconcileRecordingLifecycle` only; `handleCameraBecame*` is internal to the reconciler.
    - Trace: retention skips logged with stable reason, filename, and age seconds for auditable NVR decisions.
- Viewer analytics:
  - `viewerSessionService.js`, `viewerAnalyticsService.js`: live viewer sessions/history/aggregation; session close accepts explicit end times from HLS proxy and stale cleanup.
  - `cameraViewStatsService.js`: compact per-camera lifetime live-view counters and active-viewer public card aggregates.
  - `playbackViewerSessionService.js`: playback viewer sessions/history/aggregation.
- Playback token entitlements:
  - `playbackTokenService.js`: token/session/share/audit orchestration, public activation, stable repeat-share access codes, initial and repeat selected-camera share text names/counts via one batched camera query, default-camera metadata for public auto-selection, transaction-scoped active-session limits, throttled session touches, and bounded token maintenance cleanup.
  - `playbackTokenRuleService.js`: normalized per-camera entitlement rules, selected/all scope evaluation, and effective playback window resolution.
  - `playback_token_camera_rules`: indexed token-camera rule table; selected tokens can explicitly grant `admin_only`, while all-scope tokens exclude `admin_only` unless an enabled rule exists.
- Public growth:
  - `publicGrowthService.js`: sanitized public area/discovery read models; normalize incoming area slugs/names before DB reads so frontend area links and older raw-name URLs resolve consistently.
- Admin/settings/security:
  - `adminDashboardService.js`, `settingsService.js`, `brandingService.js`, `sponsorService.js`, `saweriaService.js`.
  - `authService.js`, `userService.js`, `apiKeyService.js`, password/session services, `securityAuditLogger.js`.
- Support utilities:
  - `thumbnailService.js`: background thumbnail stale queue, strict on-demand 3-hour thumbnail cadence, direct/on-demand generation, FFmpeg argument construction, failure backoff, and DB thumbnail metadata updates.
  - `thumbnailPathService.js`, `timezoneService.js`, `telegramService.js`, `backupService.js`, `discoveryService.js`.
  - `telegramService.js`: Telegram bot config, multi-target monitoring groups, area/camera/global routing rules, grouped camera status messages with per-camera detected timestamps, feedback messages.
  - `notificationDiagnosticsService.js`: admin-only Telegram notification preview/drill workflow; reads cameras/runtime state, uses `telegramService.js` routing helpers, and writes masked diagnostic audit rows.
  - `../utils/internalRtspTransportPolicy.js`: pure resolver for internal RTSP transport (`tcp`, `udp`, `auto`) shared by MediaMTX, recording, and thumbnail FFmpeg inputs.
  - `../utils/thumbnailStrategyPolicy.js`: pure resolver for thumbnail capture strategy (`default`, `direct_rtsp`, `hls_fallback`, `hls_only`) shared by camera writes and thumbnail generation.

## High-Risk Files

- `cameraHealthService.js`: very large and side-effect heavy. Extract only one responsibility at a time with focused tests.
- `cameraService.js`: mixes read models, writes, import/export, restore, and bulk policy. Prefer new pure helpers or repositories before adding branches.
- `recordingService.js`: facade only. Add new responsibility to a focused recording-domain module (see Recording domain above) and delegate; do not grow the facade.
- `securityAuditLogger.js`: avoid broad changes because many services depend on audit helpers.
- `settingsService.js`: settings affect frontend config, health, playback, ads, and branding.

## Cross-Service Side Effects

- Camera mutations may invalidate cache, write audit logs, update MediaMTX, update runtime/health state, and reconcile recording for recordable HLS delivery types (`internal_hls`, `external_hls`). RTSP URL, codec, delivery, enabled-state, external HLS URL, and internal RTSP transport changes must restart/start/stop FFmpeg automatically so operators do not need a manual disable/enable cycle. Live internal HLS source changes flow through `cameraSourceLifecycleService`, which sets reconnecting runtime state, refreshes the MediaMTX path lifecycle, clears per-camera health runtime caches, bumps `cameras.stream_revision`, and writes masked lifecycle events so stale active paths do not leave Camera Management stuck in reconnecting state. Internal RTSP transport and thumbnail strategy changes are opt-in and preserve existing behavior unless a camera override is set.
- Health transitions may update DB runtime state immediately, route grouped Telegram up/down notifications only after confirmation timing accepts a stable monitoring-state transition, include the original detected transition time per camera in Telegram messages, pause/resume recording, and influence thumbnail generation; runtime online signals and confirmed online transitions trigger recording lifecycle reconciliation so cameras that return after minutes or hours offline resume recording without manual toggles; public map/grid availability stays derived from camera availability state rather than Telegram alert routing, strict RTSP Telegram checks are opt-in for explicit internal camera policies/source profiles, RTSP transport remains governed by internal RTSP transport policy, and background thumbnail work must remain capped and stale-first to avoid bandwidth spikes on large camera fleets.
- Recording transitions may spawn/stop FFmpeg with explicit app-timezone `TZ`, write UTC segment rows from app-local filenames, quarantine/delete files, and write restart logs. Settings flip-to-disabled and delivery-type changes also stop active FFmpeg via the lifecycle reconciler's `stop_disabled`/`stop_unrecordable` actions — operators do not need to manually stop recording after toggling.
- Viewer session cleanup may move rows into history/archive tables and increment compact camera view counters; use the last heartbeat/HLS touch as the close time when available and keep live and playback flows separate.
- Settings updates may alter health cadence, playback policy, branding, ads, timezone, and notification behavior.

## Extraction Rules

- Prefer pure policy/helper extraction first, then service wiring.
- Keep old method signatures stable while moving internals.
- Add Header Docs to every new service/helper.
- Add or update focused tests next to existing `backend/__tests__` coverage.
- For DB work, inspect index coverage and use `transaction()` for grouped writes.
- Avoid importing route/controller modules into services.
- Migration runner hygiene: `../database/run-all-migrations.js` is import-safe, excludes aggregate scripts inside `database/migrations/`, and creates `../data/` before SQLite migration execution.
- Frontend helper boundary already present: map coordinate parsing lives in `../../frontend/src/utils/mapCoordinateUtils.js`; playback segment matching lives in `../../frontend/src/utils/playbackSegmentSelection.js`.
