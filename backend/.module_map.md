# backend/.module_map.md

## Entry Points

- `server.js`: Fastify bootstrap, security middleware chain, route registration, static thumbnails, background services, startup/shutdown.
- `routes/`: HTTP endpoint registration. Keep route-level code limited to wiring, request schemas, and route-specific proxy mechanics.
- `controllers/`: Request handlers that normalize params/body and shape `{ success, data, message }` responses.
- `services/`: Domain logic and side effects. Prefer adding focused services/helpers rather than expanding large services.
- `database/connectionPool.js`: Shared SQLite helpers: `query`, `queryOne`, `execute`, `transaction`, connection stats/close.
- `database/migrations/`: Schema/index migrations. Add forward-only migrations and run `npm run migrate`.
- `middleware/`: Security, cache, auth, input validation, rate limiting, and error handling.
- `utils/`: Small shared backend helpers such as auth cookie option derivation and camera normalization utilities.
- `__tests__/`: Backend Vitest tests. Add focused tests for service/controller/route changes.

## Domain Ownership

- Camera CRUD/import/export: `services/cameraService.js`, `controllers/cameraController.js`, `routes/cameraRoutes.js`.
- Area policy/bulk update: `services/areaService.js`, `routes/areaRoutes.js`, `cameraService.bulkUpdateArea`.
- Health/runtime state: `services/cameraHealthService.js`, `services/cameraHealthPolicy.js`, `services/cameraRuntimeStateService.js`, with Telegram routing in `services/telegramService.js`.
- Live stream orchestration: `services/mediaMtxService.js`, `services/streamService.js`, `routes/streamRoutes.js`, `routes/hlsProxyRoutes.js`.
- Viewer analytics: `services/viewerSessionService.js`, `services/viewerAnalyticsService.js`, `services/cameraViewStatsService.js`, `routes/viewerRoutes.js`.
- Public growth pages: `routes/publicGrowthRoutes.js`, `controllers/publicGrowthController.js`, and `services/publicGrowthService.js` expose sanitized area/trending/discovery camera read models for SEO/shareable public pages; `/api/public/*` is whitelisted from API-key validation as read-only public API surface.
- Playback analytics: `services/playbackViewerSessionService.js`, `routes/playbackViewerRoutes.js`.
- Recording lifecycle: `services/recordingService.js`, `recordingProcessManager.js`, `recordingRuntimeState.js`, `recordingSegmentRepository.js`, `recordingCleanupService.js`, `recordingPlaybackService.js`.
- Playback token access: `services/playbackTokenService.js`, `controllers/playbackTokenController.js`, `routes/playbackTokenRoutes.js`, and admin token/share/audit endpoints in `routes/adminRoutes.js`.
- Settings/branding/sponsor/ads: `settingsService.js`, `brandingService.js`, `sponsorService.js`, `saweriaService.js`, related routes/controllers; public runtime reads include `/api/settings/timezone` so frontend bootstrap does not require admin auth.
- Security/audit: `authService.js`, `userService.js`, `securityAuditLogger.js`, auth/security middleware.
- Time helpers: `services/timeService.js` is the shared boundary for configured local SQL timestamps, UTC SQL timestamps, local date filters, and local SQL duration calculations.

## Critical Flows

- Startup: `server.js` registers security middleware, JWT, static thumbnails, routes, then starts MediaMTX/stream warmer/health/recording/thumbnail services.
- Admin auth: `authRoutes.js` -> `authController.js` -> `authService.js`; `utils/authCookieOptions.js` derives HttpOnly cookie options so domain and direct-IP access use correct secure/sameSite behavior.
- Public live HLS: camera request -> stream route/proxy -> MediaMTX or external proxy -> HLS proxy-owned viewer session tracking for proxied streams -> health/runtime signals.
- Admin camera mutation: route/controller -> `cameraService` -> DB write -> cache invalidation -> audit log -> MediaMTX/runtime follow-up if needed; internal RTSP transport override updates MediaMTX path protocol for that camera, and thumbnail strategy controls direct RTSP versus MediaMTX HLS capture.
- Camera health loop: `cameraHealthService.checkAllCameras()` -> probe/runtime evaluation -> `camera_runtime_state` update -> grouped Telegram routing -> recording/thumbnail transitions; background thumbnail generation reads `cameras.thumbnail_strategy`, refreshes only missing/stale thumbnails, uses a 3-hour stale window for strict on-demand internal RTSP cameras, caps per-run work, and backs off repeated failures.
- Recording start: admin/service request -> source validation -> FFmpeg args -> process manager -> segment discovery/remux -> DB segment repository; internal RTSP FFmpeg transport resolves from camera override, area default, then TCP.
- Playback stream: recording routes/controller -> `recordingPlaybackService`/segment repository -> optional playback token cookie validation -> range-capable MP4 response.
- Viewer analytics: live session tables plus compact `camera_view_stats` counters -> active public card stats and admin analytics aggregation; proxied HLS sessions close at last playlist/segment touch time to avoid inflated history durations.
- Public discovery: landing page calls `/api/public/discovery` once and receives batch sections for live-now cameras, top cameras, newest cameras, and popular areas without exposing private RTSP credentials.
- Timestamp handling: live/playback viewer sessions use configured local SQL timestamps for date filtering/history, while token/audit metadata that uses SQLite `CURRENT_TIMESTAMP` is UTC SQL. Do not parse local SQL with raw `new Date()` for durations; use `timeService.diffLocalSqlSeconds`.

## Database Rules

- Use `connectionPool.js` helpers; do not create ad hoc SQLite connections in request paths.
- Use parameterized SQL with `?`.
- Check index coverage before adding analytics, history, playback, or listing queries.
- Batch write loops when updating many cameras/sessions/segments.
- Keep live viewer tables and playback viewer tables separate.

## Stabilization Targets

- `services/cameraHealthService.js`: extract probe clients, scoring, cadence, and transition side effects.
- `services/cameraService.js`: extract import/export, restore, read models, and bulk policy normalization.
- `services/recordingService.js`: keep moving lifecycle, cleanup, segment processing, and FFmpeg concerns into focused services.
- `routes/hlsProxyRoutes.js`: extract viewer identity/session cache, external host policy, and proxy response helpers.
- `middleware/schemaValidators.js`: split by route/domain when changing validation.
- `server.js`: add Header Doc sync only when changing startup flow; avoid adding more inline health/config logic.

## Verification

- Full backend gate: `npm run migrate && npm test`.
- Focused backend test: `npm test -- <test-file>`.
- For DB-heavy changes, inspect `PRAGMA index_list(<table>)` or migration index definitions.
- For streaming/recording changes, include tests for cleanup, stale process/session behavior, and path safety.
