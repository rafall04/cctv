# MultiView Module Map

## Ownership

- `MultiViewLayout.jsx`: fullscreen multi-view shell, body scroll lock, Escape/fullscreen handling, tile layout, and per-tile status isolation.
- `MultiViewVideoItem.jsx`: one live camera tile inside multi-view. Owns HLS, FLV, MJPEG, and embed/fallback tile lifecycle, playback timeout state, viewer session lifecycle, snapshot, zoom, fullscreen, and per-tile retry/error UI.
- `VideoPopup.jsx`: single-camera public popup. More complete popup player supporting HLS plus external FLV/MJPEG/embed paths, header viewer stats, minimal detail sheet, and related camera strip; skips manual viewer session tracking for proxied HLS because the HLS proxy owns those sessions.
- `CameraDetailPanel.jsx`: minimal trust/action sheet inside `VideoPopup` with area, quality label, viewer counts, playback availability, and favorite/share/area actions; area navigation must use the centralized public-growth slug helper so popup payloads with only `area_name` still open `/area/:areaSlug`.
- `RelatedCamerasStrip.jsx`: popup-local related camera rail capped to a few compact suggestions so users can move between nearby or high-interest cameras without crowding the video context.
- `MultiViewButton.jsx`: floating launcher shown when cameras are selected.
- `ZoomableVideo.jsx`: shared video zoom/pan wrapper for HLS/FLV video elements.

## Critical Flow

- Landing camera selection uses `useLandingInteractions`, computes popup-related cameras from the current public camera payload, and caps multi-view selection through `getMaxConcurrentStreams()`.
- Multi-view selection resolves public landing camera metadata through `resolvePublicPopupCamera` before adding a tile, so tiles receive the same stream payload shape as `VideoPopup`.
- `MultiViewLayout` renders up to three `MultiViewVideoItem` tiles, owns the fullscreen modal stacking layer above public mobile docks, and staggers startup by tile index.
- `MultiViewVideoItem` routes by `getMultiViewRenderMode(camera)`: HLS uses HLS.js, FLV uses flv.js, MJPEG uses an image stream, embed/JS MPEG/custom fallback uses an iframe when an embed URL exists, and unsupported custom WebSocket streams show a deterministic error.
- `MultiViewVideoItem` uses `streamInitQueue` for every device tier so multi-view startup does not burst all HLS sources at once.
- Internal HLS tiles may see temporary 404 manifest/level errors while MediaMTX warms a source; retry these before surfacing an error.
- `VideoPopup` must reset transient playback/error state when `camera.id` changes and must mask recoverable error UI while `_stream_resolution_pending` is true so related-camera switches show loading instead of stale or premature offline/error overlays.
- `VideoPopup` cleanup must be scoped to the stream run and captured media element. Do not let cleanup for camera A call `load()`/emit media errors on camera B after a related-camera switch.
- Keep viewer sessions guarded against async unmount races: start frontend viewer sessions only after playback is confirmed, immediately stop late sessions after unmount, and do not start a frontend session for internal/proxied HLS streams because the HLS proxy owns those sessions.
- Related camera clicks must route through the same popup open handlers as normal grid/map selections so stream URL resolution and session cleanup stay standardized.

## Stability Rules

- Do not place transient `loadingStage` or `autoRetryCount` state in the main HLS effect dependency list; use refs for event handlers.
- Keep HLS cleanup idempotent and scoped per tile so one camera failure does not destroy another camera.
- Multi-view supports project-defined browser-renderable public live formats: internal/external HLS, external FLV, external MJPEG, external embed, and embed fallbacks for JS MPEG/custom WebSocket cameras. Raw custom WebSocket streams remain unsupported unless an embed fallback URL is available.

## Verification

- Focused multi-view stability: `npm test -- streamInitQueue.test.js MultiViewVideoItem.stability.test.js`
- Broader media popup check: `npm test -- VideoPopup.test.jsx CameraDetailPanel.test.jsx RelatedCamerasStrip.test.jsx`
- Frontend gate for release: `npm test && npm run build && npm run lint`
